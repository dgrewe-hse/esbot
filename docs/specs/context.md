## 1. Key Relationships & Data Flow
Understanding how these parts talk to each other is crucial for **Integration Testing**.

* **User $\leftrightarrow$ Frontend:** The user submits natural language queries. The UI must handle "Loading" states while waiting for AI responses (Performance Requirement).
* **Backend $\rightarrow$ LLM:** This is a **Dependency Relationship**. The Backend transforms a user's simple question into a "Structured Prompt" (Context + Question) for the LLM.
* **Backend $\leftrightarrow$ Database:** A **Transactional Relationship**. Every message must be logged to ensure "Session Persistence" (Functional Requirement 5).
* **LLM $\rightarrow$ Backend:** The LLM returns raw text. The Backend must then "structure" this (Functional Requirement 2) before sending it to the UI.

## 2. Analysis for Software Testing

### A. The "Oracle" Problem (Non-Determinism)
Since the LLM is non-deterministic, you cannot always predict the *exact* string it will return.
* **Test Strategy:** Use **Property-Based Testing** (e.g., checking if the response contains specific keywords) rather than exact string matching.

### B. Boundary Testing (Input/Output)
* **Input:** Test how the system handles malicious prompts ("Prompt Injection") or empty queries.
* **Output:** Test how the UI renders long-form LaTeX math or code snippets generated by the LLM.

### C. Mocking External Dependencies
For **Unit Testing** the Backend, you should not call the actual LLM (it’s slow and expensive).
* **Test Strategy:** Implement a **Mock LLM Provider** that returns predefined strings to verify that the Backend correctly saves these strings to the Database.
