# Exercise 2: Requirements and Specifications

## Scope of the Exercise

In this exercise, you will using the ESBot application description to derive requirements and to create a specification for the application according to software quality assurance principles. After this exercise, your repository should contain a requirements specification for the ESBot application.

## Learning Objectives

After completing this exercise, you should be able to:

* Identify and document functional and non-functional system requirements
* Create a quality model for an application
* Use a specification driven development tool to create a specification for an application
* Create a system context diagram and use case diagram for an application
* Write use case descriptions
* Create a system context diagram and use case diagram for an application

---

## Pre-requisites

You need to have access and/or installed the following tools:
* Markdown editor
* You should have access to the ESBot application description in the `docs/esbot.md` file.
* Specification driven development tool (e.g., [GitHub Spec Kit](https://github.com/github/spec-kit))

## Exercise 2.1 (10 Points): Requirements

During the lecture, we discussed fundamental aspects of systems engineering, particularly focusing on systematically deriving clear and precise requirements. Now, using the provided introductory description of the *ESBot* application, please identify, document, and clearly specify the functional and non-functional system requirements. Document all your findings in the `docs/spec/requirements.md` file. Your documentation should comprehensively reflect the key functionalities that *ESBot* must offer to its intended users.

## Exercise 2.2 (10 Points): Quality Model

Next, we explored quality models in our lecture, focusing on how they assist in systematically deriving quality requirements and ensuring these requirements align with the characteristics defined by the *ISO 25010 standard*. Based on the provided introductory description of *ESBot*, as well as the previously documented functional system requirements, please discuss within your group, select and document the four most relevant quality aspects for the application. For each selected quality aspect, clearly define a corresponding quality model using the methods introduced during the lecture.
Additionally, considering the context of this course, identify and explain what specific measures you believe should be implemented during the development of ESBot to guarantee the quality aspect of "testability".
Document all your findings in the `docs/spec/quality.md` file.

## Exercise 2.3 (10 Points): Specification Driven Development

During the lecture, we discussed the concept of specification driven development. Now, you should familiarize yourself with the concept and use it to create a specification for the ESBot application.

In this exercise, you should use a specification driven development tool (e.g., [GitHub Spec Kit](https://github.com/github/spec-kit)) to create a specification for the ESBot application. If you are using a different tool, please justify and document your choice.

The goal: We do want to use the tool to refine the requirements and quality models you have documented in the previous exercises and create a specification for the ESBot application.

Please document your results in the `docs/spec/spec.md` file.

## Exercise 2.4 (15 Points): System Context & Use Cases

In this exercise, you will practice creating system context diagrams and use case diagrams, as previously introduced in the lab session. Based on the provided introductory description of *ESBot*, along with the documented functional system requirements, create:

1. A clearly defined system context diagram illustrating the interactions of ESBot with relevant external actors and systems.
2. A detailed use case diagram, selecting one specific feature from the provided introductory text as your example scenario.
3. Based on your use case diagram, illustrating a set of use cases for a specific feature. Please provide an appropriate use case description. Guidance on how to properly write use case descriptions is available in the ESBot repository: `docs/templates/use_cases/README.md`

Clearly document your contents and provide concise explanations highlighting key elements and their relationships.

Please document your results in the `docs/spec/context.md` and `docs/spec/use-cases.md` files.
